# 开发计划 - BugTrack Platform (MVP)

## 总览
目标：在 **4 周** 内完成 MVP 并可交付内测（可根据团队与资源调整）。  
里程碑分为 4 个迭代，每周一个迭代（Sprint）。

---

## 迭代 0 - 准备（第 0 周, 3 天）
- 确认需求 & 数据库设计（已完成）
- 搭建项目脚手架
  - 后端：Spring Boot 3（或你的偏好）+ MyBatis-Plus
  - 前端：Vue 3 + TypeScript + ElementPlus
- 建立开发/测试环境（Docker Compose / 本地 MySQL）
- 定义 API 规范（Swagger / OpenAPI）

## Sprint 1 - 基础用户 & 项目（第 1 周）
目标：用户系统 + 项目管理基本功能
- 用户注册 / 登录 / 会话（JWT 或 Session）
- 用户表单与用户信息页
- 项目 CRUD、项目成员管理（邀请/添加/删除）
- 编写基础单元测试与接口文档

## Sprint 2 - 核心 Issue 管理（第 2 周）
目标：实现缺陷（Issue）创建与基本工作流
- Issues CRUD（创建、查看、编辑、删除）
- 状态流转逻辑（simple state machine）
- 分配与指派（assignee）
- issue_no 顺序生成策略（issue_sequence 表实现）
- 基础前端页面：Issue 列表、详情、创建页

## Sprint 3 - 协作与附件（第 3 周）
目标：评论、附件与历史记录
- 评论功能（支持@用户）
- 附件上传（集成 MinIO 或本地存储）
- issue_history 自动记录（状态变更、指派、编辑）
- 简单通知（站内消息推送）

## Sprint 4 - 测试、优化与交付（第 4 周）
目标：完善、修复、部署
- 完成 API 文档（Swagger）
- 性能与索引优化（确认关键 SQL）
- 简单 E2E 测试与手工验收
- 打包部署（Docker / jar），编写部署文档
- 内测与反馈收集

---

## 风险与缓解
- 文件上传与存储（大文件） → 限制单文件大小，优先本地小文件/MinIO。
- 状态流转复杂性 → 初期实现有限状态机，后期扩展。
- 权限边界模糊 → 采用最小权限策略，测试覆盖关键场景。

## 交付物
- 后端源码（含 DB DDL）
- 前端源码（能完成常用操作）
- `tapd_mvp.sql`、`项目需求.md`、`开发计划.md`
- 部署文档与 Postman/Swagger API 文档
